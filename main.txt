\paragraph*{R2.1 Comment:} \textcolor{Plum}{Although this work claims that non-binary (NB) polar codes provide better decoding performance and can achieve low-power consumption, it would be beneficial to include a direct comparison between the proposed method and traditional binary polar codes, particularly in terms of power consumption and throughput.}

\paragraph*{Author response:} Thank you very much for your comment, which highlights a tricky point in the paper and points out an ambiguous statement.

First, in terms of complexity, binary code will always be more efficient than non-binary code in terms of decoding power consumption and throughput. For example, in the case of binary polar codes, the f-function requires only a single operation, resulting in a complexity of O(1), whereas for NB-p codes over GF(q), the complexity per bit is O(q). For instance, in 2015, \cite{LeG14} reported a software implementation of binary polar codes achieving a decoding throughput greater than 2 Gbit/s per CPU core. From the polar decoding perspective alone, there is absolutely no advantage in using NB-p codes over binary polar codes in terms of throughput and energy if the coding gain is not considered.

However, when considering the energy and throughput of the transmission itself, the picture changes. In fact, the performance of coded modulation significantly outperforms that of bit-interleaved coded modulation (see \cite{Liva17}, for example). The figure \ref{fig:josheph}, extracted from Jocelyn Bourduge’s PhD thesis \cite{Bou24}, shows that the use of coded modulation (and thus, a non-binary code) provides a 2 to 3 dB gain at low SNR compared to BICM modulation (which uses a binary code). This capacity gain translates directly into transmission throughput and/or energy savings (the same amount of information can be transmitted faster for a fixed SNR).

\subsubsection*{\emph{\textcolor{blue}{V.B. Comparison with related works}}}

\emph{\textcolor{blue}{To the best of our knowledge, no software implementations dedicated to the decoding of NB-p codes have been reported in the literature, preventing a direct comparison. Nevertheless, several efficient software implementations have been developed for instance for binary polar code decoding \cite{Cas15,LeG14,Sar14,Gia18} and for non-binary LDPC codes \cite{Fer22,LeG20}, which represent the closest works to our contribution. It is important to emphasize that the non-binary polar decoders proposed in this work, which rely on the SC-FFT decoding algorithm, achieve higher error-correction performance than conventional binary polar decoders based on SC \cite{Cas16} and SC-SCAN \cite{LeG18}. This performance gain, however, comes at the cost of a higher computational complexity, as discussed in Section I.}}

\emph{\textcolor{blue}{In the literature, binary SC decoders \cite{LeG14} have been shown to achieve very high throughput, exceeding 2 Gbps per core on platforms similar to P3 for code lengths $N<2048$ using a single processing core. However, such high throughput, which naturally results in a reduction of the energy consumption per decoded bit (by approximately one order of magnitude), is achieved at the expense of a degradation in error-correction performance of about 2 to 3 dB.}}

\emph{\textcolor{blue}{In parallel, more powerful binary polar code decoders in terms of error-correction capability have been proposed to partially compensate for SNR gaps. These decoders are typically based on the SCL algorithm \cite{Tal11}. Several software implementations have been reported in the literature \cite{Leo19,Liu22}. However, these implementations generally rely on a conventional SC decoding stage and invoke SCL decoding only when the initial SC decoding fails. This strategy artificially increases the average throughput at high SNR values (e.g., $>3.0$ dB), since the computationally intensive SCL decoding is rarely executed. Nevertheless, such approaches exhibit significantly reduced throughput at low SNR, where the execution of the SCL decoder becomes necessary \cite{Leo19,Liu22}. In practice, the reported throughput on CPU and GPU platforms does not exceed a few Mbps per core, despite the use of more complex and power-hungry architectures (e.g., an RTX3090 GPU \cite{Liu22}).}}

\emph{\textcolor{blue}{A similar observation has been reported in the context of non-binary LDPC codes \cite{LeG20}. In such systems, the achievable throughput directly depends on the average number of decoding iterations required for convergence, which is itself strongly correlated with the SNR. Experimental evaluations performed on the P3 platform using the software implementations described in \cite{LeG20} show that, for finite fields $GF=\{16,64,256\}$, a code rate of 50\%, and up to ten decoding iterations, the achieved throughput ranges from 1 to 6 Mbps in single-core mode at 0 dB when all ten iterations are effectively required. These throughput values are more than one order of magnitude lower ($>10\times$) than those reported in Section V-C of this paper, which consequently implies a similar increase in the energy consumption per decoded bit.}}

\emph{\textcolor{blue}{In the proposed approach, the computational complexity of the decoder is independent of the SNR, including in low-SNR regimes ($<0$ dB). Although a direct comparison remains difficult due to the open and heterogeneous evaluation frameworks considered in the literature, the proposed solution enables operation in low-SNR conditions while maintaining a consistently high throughput regardless of the channel conditions.}}

\begin{revisionenv}
\emph{\textcolor{blue}{For the LoRa modulation, a simpler, but equivalent, demodulation algorithm is usually applied. A LoRa symbol is modulated using a chirp with a frequency that varies linearly over time. Considering a given spreading factor SF, the $m^{th}$ symbol $x_m$ of the $2^{SF}$ dictionary codeword can be represented in digital by $x_m$, a vector of size $2^{SF}$ given as}}

\begin{equation}
x_m(k) = e^{ 2\pi j \left( \frac{k^2}{2 \cdot 2^{\text{SF}}} - \frac{m k}{2^{\text{SF}}} \right)}, \: \: k = 0, 1, \dots, 2^{\text{SF}} - 1.

r(m) &= \sum_{k=0}^{2^{SF}-1} y(k)e^{- 2\pi j \left(\frac{k^2}{2.2^{SF}} - \frac{m k}{2^{\text{SF}}} \right)},\\
     &= \sum_{k=0}^{2^{SF}-1} \left(y(k)e^{- 2\pi j \frac{k^2}{2.2^{SF}}}\right) e^{2\pi j \frac{m k}{2^{\text{SF}}} }\\
\label{eq:CSS_demos}
\end{align}

\emph{\textcolor{blue}{which is equal to the $m^{th}$ terms of the Inverse Fast Fourier Transform (IFFT) of the terms by terms product of $y$ and $\text{conj}(x_0)$ multiplied by a factor $2^{SF}$.}}

\paragraph*{R3.3 Comment:} \textcolor{Plum}{SPC Node Convergence and SNR Wall: Given that the SPC approximation introduces a $\sim$0.1 dB loss, the authors should provide a convergence or "SNR Wall" analysis across different code rates ($R = 1/4, 1/2, 3/4$). It is critical to ensure that this approximation does not lead to an irreducible error floor at extremely low Frame Error Rates (FER), which is a vital metric for mission-critical IoT applications.}
\\

\paragraph*{Author response:} Thank you very much for your insightful comment. It can be addressed in two ways: theoretically and through Monte-Carlo simulation.
The theoretical approach would involve first studying the conditions under which the SPS approximation fails while the exact SPS does not. Next, it would require assessing the probability of such events occurring (as a function of parity check size and SNR), and finally, evaluating the impact of these errors on overall performance. While a theoretical approach is feasible, it is not a straightforward approach.
On the other hand, Monte-Carlo simulation directly allows us to verify the absence of an error floor down to very low FER values (e.g., $10^{-7}$ in the revised version of the paper). Nevertheless, it is worth noting that the software used in the initial submission did exhibit an error floor around a FER of $10^{-5}$. However, this error floor was not caused by the SPS approximation but rather by numerical instability in finite-length computations, which resulted in some probability values becoming zero or negative.
This issue has been resolved by introducing a clipping function after computing each probability vector, for example, $P= \max(P, 10^{-10})$. Naturally, this function has a negligibleslight impact on decoding throughput, and all the results in this revised version account for this modification.

\paragraph*{Modifications:} We thus add the following text in the section III.A of the revised paper version.

\emph{\textcolor{blue}{In practice, even with floating-point arithmetic, the inverse Hadamard transform can generate near-zero or even small negative probability values due to finite precision effects, particularly when subtracting nearly equal numbers.
These spurious zeros, if left unclipped, can accumulate through successive decoding layers and eventually cause an error floor at high SNRs.
To prevent this, we apply a simple clipping operation after every inverse transform.
Any probability value smaller than a threshold $\varepsilon = 10^{-10}$ is replaced by $\varepsilon$.
The threshold $\epsilon$ is chosen sufficiently small to have no measurable impact on performance, while being large enough to flush out underflown values.
Empirical results prove that this clipping eliminates the error floor entirely, up to a frame error rate of $10^{-8}$.
In the following, we denote by $\bar{\mathcal{H}}$ the inverse Hadamard transform that includes this clipping.}}


\paragraph*{R3.4 Comment:} \textcolor{Plum}{In Tables I and II, the complexity comparison should distinguish between the computational "cost" of multiplications versus the additions/subtractions in the FWHT. In modern SIMD software execution, these operations have different latencies and throughputs. A more nuanced discussion on how these weights impact actual execution time would improve the clarity of the complexity analysis.}

\paragraph*{Author response:} Thank you for this valuable remark, which we had not analyzed in detail until now. At first glance, when reading your comment, we had the same intuition as you. However, after examining the official documentation as well as previous studies related to Intel and Arm architectures, we reached the opposite conclusion. Indeed, it is true that integer addition and multiplication operations (int32b) exhibit different latencies and throughputs. According to the official documentation from Intel (Intel Intrinsics Guide), for 32-bit integer operations on the Skylake architecture, an addition has a latency of 1 cycle and a throughput of 0.33 cycles per instruction, whereas a multiplication has a latency of 5 cycles and a throughput of 0.5 cycles per instruction.

For floating‑point instructions the picture is different. Referring again to Intel’s official documentation \cite{intel:guide}, the \emph{vaddps} and \emph{vmulps} instructions exhibit identical characteristics on Skylake: a latency of 4 cycles and a throughput of 0.5 instruction per cycle. On Alder Lake architectures, however, the latency of \emph{vaddps} becomes roughly half that of \emph{vmulps}. This Alder Lake behaviour appears to be a special case \cite{Abe19}. The theoretical observations are confirmed by experimental results reported in \cite{Abe19}. Benchmarks show that the execution times and throughput of \emph{vaddps} and \emph{vmulps} are essentially identical on most Intel platforms (Skylake‑X, Coffee Lake, Tiger Lake, Rocket Lake) but also on AMD Zen ones.

Regarding ARM architectures, available data are more limited because vendors typically provide few micro‑architectural details for their custom cores. This is especially true for Apple Silicon processors (Apple M4). Nevertheless, performance analysis of a Cortex‑M7 core also reveals identical latencies and throughputs for floating‑point addition and multiplication instructions \cite{arm:guide}.

\paragraph*{Modifications:} It remains difficult to generalise these observations across all contemporary processor cores. \paragraph*{R3.7 Comment:} \textcolor{Plum}{The proposed DPF-FSC algorithm utilizes a "dual-domain buffer" strategy to minimize redundant transforms. \paragraph*{Author response:} Thank you for highlighting the lack of precision in the original manuscript. The peak memory footprint of the dual‑phase decoder is identical to that of the decoder handling only probabilities (the original algorithm). The memory region holding exchanged messages—comprised of $N$ slots each containing $P=\log_2(q)$ elements with the same data size regardless of whether probabilities or frequencies are stored. Memory cost is therefore identical in both cases, since each probability or frequency value is represented by a 32‑bit floating‑point number. During decoding, both approaches require two memory buffers of $P$ elements to perform domain conversions (probability from/to frequency) before the multiplications. In conclusion, the dual‑phase approach minimizes domain conversions during decoding without increasing runtime memory complexity. Note that low-power MCUs implementation with RF shields is part of our future works.

\begin{revisionenv}
